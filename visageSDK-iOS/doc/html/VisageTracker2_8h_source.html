469  /* Get facial feature points estimated by the tracker.
470  *
471  * The feature points are identified
472  * according to the MPEG-4 standard, so each feature point is identified by its group and index. For example, the tip of the chin
473  * belongs to group 2 and its index is 1, so this point is identified as point 2.1. The identification of all MPEG-4 feature points is
474  * illustrated in Figure 2 on Page 8 of the <a href="../MPEG-4 FBA Overview.pdf">MPEG-4 Face and Body Animation Introduction</a>.
475  *
476  * Certain feature points, like the ones on the tongue and teeth, can not be reliably detected so they are not returned
477  * and their coordinates are always set to zero. These points are:
478  * 3.5, 3.6, 6.1, 6.2, 6.3, 6.4, 7.1, 9.8, 9.9, 9.10, 9.11, 10.1, 10.2, 10.3, 10.4, 10.5, 10.6, 11.4, 11.5, 11.6
479  * Several other points are estimated, rather than accurately detected, due to their specific locations. Examples of such points
480  * are 11.4 (usually hidden in the hair), hairline points (10.1 - 10.3) as well as cheeck points 5.1 - 5.4 which can not be precisely defined due to lack of
481  * features on the cheek.
482  *
483  * The resulting feature point coordinates are returned in form of an FDP object. This is a container clas used for storage of MPEG-4 feature points.
484  * It provides functions to access each feature point by its group and index and to read its coordinates. Note that FDP can store 3D points but
485  * in case of 2D return feature points only the x and y coordinates of each point are used.
486  *
487  * In case of absolute 3D coordinates (relative = false), the coordinate system is such that when looking towards the video image, the direction of x is to the
488  * left, y iz up, and z points into the image, away from the viewer. The global origin (0,0,0) is placed at the camera.
489  *
490  * In case of relative 3D coordinates (relative = true), the coordinate system is such that when looking towards the video image, the direction of x is to the
491  * right, y iz up, and z points out of the image, towards the viewer. The origin (0,0,0) is placed at the left eye of the person (i.e. the eye that is on the right in the image).
492  *
493  * The 2D coordinates are normalised to image size so that the lower left corner of the image has coordinates 0,0 and upper right corner 1,1.
494  *
495  * @param initial if true, function returns the feature points from the initial frame of video, otherwise it returns the latest estimated points.
496  * @param relative if true, function returns the coordinates relative to the head (i.e. coordinates in the coordinate system of the head), otherwise it returns global coordinates. this parameter is ignored if 2D coordinates are requested.
497  * @param _3D if true, the function returns 3D coordinates of feature points. Otherwise, it returns 2D coordinates normalised to image size so that the upper left corner of the image has coordinates 0,0 and lower right corner 1,1.
498  */
499  FDP *getFeaturePoints(bool initial, bool relative, bool _3D);

585  /* Get the translation of the face (head).
586  *
587  * This function returns the current estimated translation of the head,
588  * either relative to its initial position in the first video frame or absolute (i.e. relative to the camera position).
589  * A typical application using relative translation is character animation, while absolute translation is usefull for example in AR type of applications.
590  *
591  * Translation is expressed with three coordinates x, y, z.
592  * When looking towards the video image, the direction of x is to the
593  * left, y iz up, and z points forward into the image.
594  *
595  * If relative is TRUE, the function returns the translation of the head relative to its
596  * position in the first video frame. The coordinates are normalised with respect to the eye separation
597  * distance of the tracked face. This means that the value of 1
598  * corresponds to the translation equal to the distance between the
599  * person's eyes.
600  *
601  * If relative is TRUE, the function can return compensated or uncompensated
602  * translation depending on the value of the parameter compensated.
603  * Uncompensated translation is the raw translation estimated by the tracker.
604  * Compensated translation values take into account the fact that the tracker
605  * uses a relatively flat face model for tracking, so the center of rotation of this model
606  * is in the front area of the head, while the anatomical center
607  * of rotation is behind, in the base of the neck. Therefore, when the rotation
608  * is applied to a 3D head model with anatomically correct center of rotation, the
609  * face naturally translates as well. When this translation is compounded with
610  * the translation values obtained from the tracker, the total resulting translation
611  * is exaggerated. To avoid this exaggerated translation of the animated head,
612  * the translation can be compensated. The compensation algorithm
613  * estimates how much the translation would be exaggerated, and makes it that much
614  * smaller. The compensated translation can directly be applied to animated head
615  * models that use the neck base as the center of rotation, and is expected to give
616  * better results than the uncompensated translation. The compensation can be fine-tuned
617  * using the @ref translation_compensation_factor attribute.
618  *
619  * If relative is FALSE, the function returns the position of the head
620  * relative to the camera, i.e. the absolute head position in the camera space. The parameter "compensated"
621  * is ignored.
622  * If the value set for the camera
623  * focal length in the <a href="doc/VisageTracker Configuration Manual.pdf">tracker configuration</a> file
624  * corresponds to the real camera used the scale of the coordinates shall be correct, in meters; otherwise the scale of the translation values is not known, but the relative values are still correct (i.e. moving towards the camera results in negative values of z coordinate).
625  * The absolute translation, rotation returned by getFaceRotation() and the camera focus value FBFT::f together form the 3D coordinate system of the head in its current position
626  * and they can be used to align 3D rendered objects with the head for AR or similar applications. The relative facial feature coordinates returned by getFeaturePoints()
627  * can be used to align rendered 3D objects to the specific features of the face, like putting virtual eyeglasses on the eyes.
628  *
629  *
630  * @param relative if true, function returns relative translation from the position in the initial video frame; otherwise it returns absolute translation (from the camera).
631  * @param compensated if true, and if "relative" is also true, function returns compensated translation, otherwise it returns uncompensated translation.
632  * @return pointer to the array with the x, y, z translation values.
633  *
634  * @see translation_compensation_factor
635  * @see getFaceRotation()
636  * @see getFeaturePoints()
637  * @see FBFT
638  */
639  float *getFaceTranslation(bool relative, bool compensated);

642  /* Get the textured 3D model of the face (head) in current pose.
643  *
644  * This method is currently experimental and has not been fully tested.
645  *
646  * This method returns the 3D model internally used by the tracker. The model is fitted in 3D to the
647  * face in the current video frame. The model is a single textured 3D triangle mesh.
648  *
649  * The texture of the model is the current video frame. This means that, when the model is drawn using the correct
650  * perspective (defined by focal length f, image width and height and model rotation and translation), it exactly recreates the facial part of the image. This can be used, for example, to
651  * cut out the face from the rest of the image (NOTE: if this is the desired effect, the model with closed mouth should be used otherwise when the mouth opens a hole appears.
652  * The model candide3-ClosedMouth.wfm is provided for this purpose and should simply be specified instead of the default Candide3.wfm in the model_filename statement in the tracker configuration file).
653  *
654  * There are multiple potential uses for this function. Some ideas include, but are not limited to:
655  *
656  * - Draw the model into the Z buffer to achieve correct occlusion of virtual objects by the head in AR applications.
657  * - Use texture coordinates to cut out the face from the image.
658  * - Draw the 3D model from a different perspective than the one in the actual video.
659  * - Insert the model into another video or 3D scene.
660  *
661  * Note that smoothing parameters are not applied to any values returned by this function.
662  *
663  * @param trans 3D translation of the head (equivalent to calling getFaceTranslation(false, false) except that this function returns original values without any smoothing, while getFaceTranslation() returns smoothed values if smoothing filter is on).
664  * @param rot 3D rotation of the head (equivalent to calling getFaceRotation(false) except that this function returns original values without any smoothing, while getFaceTranslation() returns smoothed values if smoothing filter is on).
665  * @param f Camera focal lenght, to be used in setting the perspective if the 3D model has to be rendered correctly aligned with current video image.
666  * @param img_width Width of the current video image, which is also the texture for the 3D model.
667  * @param img_height Height of the current video image, which is also the texture for the 3D model.
668  * @param tex_img The current video image, which is also the texture for the 3D model. IplImage is the image storage class from OpenCV, please refer to OpenCV documentation for details of accessing its data members; the basic members are the size of the image (frame->width, frame->height) and the pointer to the actual pixel data of the image (frame->imageData).
669  * @param n_vert Number of vertices in the 3D model.
670  * @param vert List of vertex coordinates of the 3D model, x y and z coordinate for each vertex.
671  * @param n_tri Number of triangles in the model.
672  * @param tri Triangles list Each triangle is described by three indices into the list of vertices vert. Counter-clockwise convention is used for normals direction.
673  * @param tex_coord Texture coordinates for the model, a pair of u, v coordinates for each vertex.
674  * @return true on success, false if the model can not be returned because the tracker is not active or the face is currently not detected in the video frame.
675  *
676  * @see getFaceTranslation(), getFaceRotation), VisageSDK::FBFT::f
677  */
678  bool getFaceModel(const float* &trans, const float* &rot, float &f, int &img_width, int &img_height, const IplImage* &tex_img, int &n_vert, const float* &vert,int &n_tri, const int* &tri, const float* &tex_coord);
